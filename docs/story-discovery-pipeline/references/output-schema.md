## Storage Architecture

The pipeline outputs map to three storage tiers depending on how they're used by the site.

### Tier 1: Weaviate (persistent, indexed, searchable)

These objects are stored as Weaviate collections with vector embeddings and cross-references.
They power search, entity pages, and the "discover connections" layer.

| Pipeline Output            | Weaviate Collection          | Vectors                                                                     | Cross-References                         | Notes                                                                                                                                                    |
| -------------------------- | ---------------------------- | --------------------------------------------------------------------------- | ---------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **SegmentIndex**           | `Chunks` (extends existing)  | `transcription_vector` (text), `question_vector` (identity-level questions) | `belongsToTestimony`, `mentionsEntities` | The question_vector is the key addition — it enables searching by what question a segment answers, not just what words it contains                       |
| **QuestionConstellations** | Properties on `Chunks`       | Embedded as part of `question_vector`                                       | —                                        | Facts/feelings/identity questions stored as text properties; identity-level questions embedded as the second named vector                                |
| **EntityGraph**            | `Entities` (new collection)  | `description_vector` (for semantic entity search)                           | `mentionedInChunks`, `relatedEntities`   | Entity type, canonical form, variants, linked data URIs, internal IDs stored as properties. Relationships stored as cross-references with qualifiers.    |
| **NarrativeExtraction**    | Properties on `Chunks`       | —                                                                           | —                                        | Values, emotional_charge, tensions, metaphors stored as JSON text properties on existing chunks. Enriches filtering but doesn't need its own collection. |
| **ConflictMap**            | `Conflicts` (new collection) | —                                                                           | `involvesChunks`, `involvesTestimonies`  | Conflict type, scope, description, the shared question, unresolved flag. Cross-refs to the chunk pairs involved.                                         |

### Tier 2: Weaviate + Editorial CMS (persistent, editorially managed)

These objects are generated by the pipeline but reviewed and potentially edited by humans
before they become public-facing. They need a CMS-like workflow (draft → review → publish).

| Pipeline Output         | Storage                                          | Notes                                                                                                                                                                                                                      |
| ----------------------- | ------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **QuestionThreads**     | `QuestionThreads` collection in Weaviate         | Theme label, question, question level, convergence type. Cross-refs to answer chunks. Editorial team reviews theme labels and may merge, rename, or suppress threads before publishing. Theme labels power the word cloud. |
| **NarrativeThreads**    | `Threads` collection in Weaviate or separate CMS | Merged threads from all three Pass 2 stages. Thread type, dramatic question, theme label, strength score. Cross-refs to supporting QuestionThreads and chunks.                                                             |
| **StorylineCandidates** | Separate CMS or JSON storage                     | Editorial compositions that reference threads and chunks but have their own narrative structure. Drafts until published. Question sequences, evidence maps, gap analysis.                                                  |
| **CompositionMap**      | Separate CMS or JSON storage                     | The editorial arrangement for a published storyline. Section order, included segments, excluded tangential material. Changes as editors work with it.                                                                      |

### Tier 3: Ephemeral (computed, not stored)

These are intermediate analytical artifacts used during pipeline execution. They inform
the outputs in Tiers 1 and 2 but don't need to persist.

| Pipeline Output                               | Notes                                                                                                                                                |
| --------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| Multi-frame analysis outputs (Pass 2 Stage 2) | Working notes — patterns surfaced by each of the five interpretive frames. Useful during analysis, consumed during thread synthesis, then discarded. |
| Pass 2 Stage 3 intermediate analysis          | Transformation detection and meta-thread detection working artifacts. The resulting threads are persisted; the analysis process is not.              |

### Weaviate Schema Extensions

The existing ts-portal-american-stories schema has two collections: `Testimonies` and `Chunks`.
The pipeline extends this with:

**New named vector on Chunks: `question_vector`**
In addition to the existing `transcription_vector`, each chunk gets a second named vector
embedding the identity-level questions it answers. This enables dual search: a query can
match against what was said (transcription_vector) OR what question the segment answers
(question_vector). The Holocaust Museum technique, operationalized.

**New collection: `Entities`**
Properties: canonical_form, entity_type, variants (text array), linked_data_qid,
linked_data_url, internal_id, context_summary, transcription_notes.
Cross-references: mentionedInChunks (many-to-many with Chunks), relatedEntities
(self-referential, with relationship_type qualifier).

**New collection: `Conflicts`**
Properties: conflict_type, scope, description, shared_question, significance,
unresolved (boolean).
Cross-references: involvesChunks (to Chunk pairs), involvesTestimonies.

**New collection: `QuestionThreads`**
Properties: theme_label, thread_question, question_level, source_count, convergence,
published (boolean — editorial gate).
Cross-references: answeredByChunks (to Chunks that answer this question).

**How the site queries Weaviate:**

- **Homepage word cloud**: Query all published QuestionThreads, aggregate by theme_label,
  size by source_count.
- **Theme click**: Filter QuestionThreads by theme_label, return the question(s) and
  their answeredByChunks cross-references grouped by testimony.
- **Search**: Dual vector search — query against both transcription_vector and
  question_vector on Chunks, merge and rank results, tag each result with which
  vector space matched (Facts/Feelings/Identity badge on search results).
- **Entity page**: Query Entities by canonical_form, follow mentionedInChunks
  cross-references, group by testimony. Query Conflicts that involvesChunks
  overlapping with this entity's mentions.
- **Testimony page chapters**: Query Chunks for a testimony, ordered by start_time,
  with their NarrativeExtraction properties and entity cross-references.
- **Connected stories**: For a given testimony's chunks, find other testimonies that
  share entities (via Entity cross-refs) or QuestionThreads (via answeredByChunks).
- **Storyline page**: Retrieve the editorial StorylineCandidate, resolve its chunk
  references from Weaviate, render in the composed order.

## SegmentIndex

Produced by: Pass 0, Step 1
One per source.

```json
{
  "source_id": "string — slug derived from narrator name",
  "source_metadata": {
    "narrator": "string",
    "interviewer": "string | null",
    "date_recorded": "ISO date string | null",
    "location": "string | null",
    "duration": "HH:MM:SS string | null",
    "project": "string — name of the oral history project",
    "question_set": "string | null — reference to interview guide used"
  },
  "segments": [
    {
      "segment_id": "string — {source_slug}__seg_{NNN}",
      "speaker": "string — speaker name as it appears in transcript",
      "speaker_role": "NARRATOR | INTERVIEWER | OTHER",
      "start_time": "HH:MM:SS | null",
      "end_time": "HH:MM:SS | null",
      "char_start": "integer | null — for untimestamped sources",
      "char_end": "integer | null",
      "text": "string — exact original text",
      "segment_summary": "string — one-sentence neutral description"
    }
  ],
  "total_segments": "integer",
  "total_duration": "HH:MM:SS | null"
}
```

## EntityGraph

Produced by: Pass 0, Step 2
One per source, then merged into a collection-level graph.

```json
{
  "source_id": "string",
  "entities": [
    {
      "entity_id": "string — canonical slug or internal ID",
      "canonical_form": "string",
      "entity_type": "PERSON | PLACE | EVENT | DATE | INSTITUTION | CULTURAL_ITEM",
      "variants": ["string — all forms found in transcript"],
      "linked_data": {
        "wikidata_qid": "string | null",
        "wikipedia_url": "string | null",
        "geonames_id": "string | null",
        "description": "string | null — brief description from linked source"
      },
      "internal_id": "string | null — for entities with no linked data match",
      "context_summary": "string — one sentence on this entity's role in this source",
      "mentions": [
        {
          "segment_id": "string",
          "entity_text": "string — exact text as it appears",
          "context_role": "string — role in this specific segment",
          "confidence": "HIGH | MEDIUM | LOW"
        }
      ],
      "transcription_notes": [
        {
          "variant": "string — the potentially erroneous form",
          "likely_correct": "string",
          "reason": "string"
        }
      ]
    }
  ],
  "relationships": [
    {
      "relationship_id": "string",
      "subject": "string — entity canonical form",
      "subject_id": "string — entity_id",
      "relationship": "string — FAMILY_OF | REPRESENTED | INTERNED_AT | LIVED_IN | WORKED_AT | FOUGHT_FOR | CREATED | LED | TAUGHT | SAVED | OPPOSED | COLLABORATED_WITH | INFLUENCED | PRECEDED | FOLLOWED",
      "object": "string — entity canonical form",
      "object_id": "string — entity_id",
      "qualifier": "string — additional context for the relationship",
      "grounding_quote": "string — exact text from transcript",
      "segment_id": "string",
      "confidence": "HIGH | MEDIUM | LOW"
    }
  ]
}
```

### Collection-Level Entity Graph

When merging per-source graphs into a collection graph, reconcile entities that appear
across multiple sources. The same person, place, or event referenced in different
transcripts should resolve to a single entity node with mentions from all sources.

```json
{
  "collection_id": "string",
  "entities": ["...same as above but with mentions spanning multiple sources"],
  "relationships": ["...same as above, from all sources"],
  "cross_source_connections": [
    {
      "entity_id": "string",
      "sources": ["string — list of source_ids that mention this entity"],
      "note": "string — significance of this entity appearing across sources"
    }
  ]
}
```

```json

      "confidence": "HIGH | MEDIUM | LOW",
      "note": "string | null — explanation of ambiguity"

    "end_charge": "positive | negative | neutral",

        "at": "string — quote where the turn happens",
        "from": "string — emotional state before",
        "to": "string — emotional state after"
```

## QuestionThreads

Produced by: Pass 2, Stage 1
One per collection. This is the primary cross-source discovery output — each thread
IS a question, and its value is an array of excerpts from across the collection.

```json
{
  "collection_id": "string",
  "question_threads": [
    {
      "thread_id": "string",
      "thread_question": "string — the shared question, written as a human would ask it",
      "theme_label": "string — 1-3 words summarizing this question as a browseable theme (word cloud, tag, navigation pill)",
      "question_level": "FACTS | FEELINGS | IDENTITY",
      "answers": [
        {
          "source_id": "string",
          "segment_id": "string",
          "excerpt": "string — exact words from the transcript, not paraphrased",
          "narrator": "string — speaker's name",
          "context": "string — one sentence of place, time, situation"
        }
      ],
      "source_count": "integer",
      "convergence": "AGREE | DIVERGE | CONTRADICT",
      "conflicts": ["string — conflict_ids from ConflictMap"]
    }
  ]
}
```

## NarrativeThreads

Produced by: Pass 2 (all three stages merged)
One per collection. Includes question threads, frame-based threads, transformation
threads, and meta-threads, deduplicated and ranked.

```json
{
  "collection_id": "string",
  "threads": [
    {
      "thread_id": "string",
      "thread_name": "string — evocative short name",
      "thread_type": "QUESTION | TENSION | TRANSFORMATION | META — which stage discovered it",
      "dramatic_question": "string — the Rabiger dramatic question; if you cannot form one, this is a topic cluster not a thread",
      "core_tension": "string — the opposing forces (for TENSION type)",
      "transformation": "string — the before → after (for TRANSFORMATION type, null otherwise)",
      "question_thread_ids": ["string — IDs of QuestionThreads that feed into this thread"],
      "sources_involved": ["string — source_ids"],
      "supporting_segments": ["string — segment_ids"],
      "temporal_arc": "string — INTENSIFIES | RESOLVES | CYCLIC | OPEN",
      "key_voices": ["string — narrator names"],
      "system_dynamic": "string — Meadows-lens description of underlying system",
      "frames_that_surfaced_it": ["string — which analytical frames found this"],
      "robustness": "integer — number of frames (1-5)",
      "conflicts": ["string — conflict_ids central to this thread"]
    }
  ]
}
```

## CompositionMap

Produced by: Pass 4
One per selected storyline.

```json
{
  "storyline_id": "string",
  "roles": [
    {
      "segment_id": "string",
      "source": "string",
      "rabiger_function": "ADVANCES | COMPLICATES | RESOLVES — the coarse Rabiger classification",
      "primary_role": "SETS_THE_WORLD | INTRODUCES_TENSION | ESCALATES | COMPLICATES | TURNS | RESOLVES_OR_REFRAMES | PROVIDES_EVIDENCE | REVEALS_THE_SYSTEM | ADDS_TEXTURE | TANGENTIAL",
      "secondary_role": "string | null",
      "placement": "string — which question in the sequence this serves",
      "quote": "string — single most powerful line for this storyline",
      "note": "string — why this segment serves this role"
    }
  ],
  "composition_sequence": [
    {
      "section_question": "string — from the question_sequence",
      "segments_in_order": ["string — segment_ids arranged narratively"]
    }
  ],
  "unused_tangential": [
    {
      "segment_id": "string",
      "source": "string",
      "note": "string — why this was excluded"
    }
  ]
}
```

## Collection Manifest

Top-level object that ties everything together.

```json
{
  "collection_id": "string",
  "collection_name": "string",
  "project": "string — e.g., American Stories",
  "sources": ["string — list of source_ids"],
  "total_segments": "integer",
  "total_entities": "integer",
  "total_relationships": "integer",
  "total_conflicts": "integer",
  "total_threads": "integer",
  "total_storyline_candidates": "integer",
  "pipeline_version": "string",
  "processed_at": "ISO datetime",
  "interview_question_set": "string | null — reference to the question guide used"
}
```
